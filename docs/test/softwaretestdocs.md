## 测试阶段

从代码最小单元到整个完整系统的逐层验证。

### 单元测试

软件测试的基础层级，旨在对软件中**最小的可测试单元**（通常是单个函数、方法或类）进行独立验证，确保其逻辑在隔离状态下运行正确。

#### 核心特征

- **粒度最小**：聚焦于代码的最基本组成部分。
- **高度隔离**：测试时不依赖数据库、网络或外部文件系统。对于外部依赖，通常使用 **Mock（模拟对象）** 或 **Stub（桩对象）** 来替代。
- **快速反馈**：单个用例通常在毫秒级完成，支持开发者在编写代码时频繁运行。
- **白盒测试**：开发人员需要了解代码的内部结构，编写测试用例来覆盖不同的执行路径。

#### 主流测试框架和工具

- **Java**: JUnit 5 (行业标准)、TestNG。
- **Python**: Pytest (功能强大且流行)、Unittest。
- **JavaScript**: Jest (Facebook 出品，内置 Mock)、Vitest (速度极快)。
- **C# / .NET**: xUnit、NUnit。
- **Mock 工具**: Mockito (Java)、unittest.mock (Python)。

### 集成测试

在单元测试之后进行的测试阶段，其核心目标是验证**不同模块或组件之间的接口交互**是否正确。即使单个模块运行正常，组装后也可能因参数传递、通信协议或逻辑冲突产生缺陷。

#### 核心特征

集成测试不同于孤立的单元测试，也不同于全面的系统测试，其特征如下：

- **关注点在于接口**：不关心模块内部逻辑，只关注模块间的数据传输、参数匹配、调用顺序及协议通信。
- **低耦合模块的粘合**：验证多个开发者编写的不同模块组装在一起时，是否会因为理解偏差或全局变量冲突导致崩溃。
- **环境依赖性强**：通常需要真实运行环境（如数据库、缓存、第三方 API）或半真实的模拟环境，无法像单测那样完全脱离外部资源 。
- **承上启下**：位于单元测试之后、系统测试之前，是发现**架构设计缺陷**和**性能瓶颈**的关键阶段

#### 主流测试工具类别

API接口集成:Postman/SoapUI。验证 RESTful、SOAP 或 GraphQL 接口间的请求与响应是否正确

Spring框架:Spring Boot Test。利用其丰富的注解（如 `@SpringBootTest`）在 Spring 框架内模拟真实的依赖注入与交互

前端与后端集成:Cypress / Playwright。模拟用户真实操作，验证前端界面与后端接口的端到端（E2E）集成链路

性能集成: JMeter。用于测试多个接口组装后的业务流在高并发下的性能表现

自动化框架:Pytest (Python)。 通过插件（如 `pytest-requests`）灵活编写涵盖多个模块调用的集成脚本

### 系统测试

在集成测试通过后进行的环节，它将整个软件看作一个**完整的实体**。此时，测试不再关注内部代码逻辑（黑盒模式），而是模拟真实用户，在实际运行环境下验证软件是否满足了《需求规格说明书》的所有要求。

#### 核心特征

- **黑盒测试（Black-box）**：测试人员无需了解代码结构，仅通过输入和输出来验证功能。
- **全路径验证**：涵盖从用户登录、业务操作到数据退出的**完整业务流**。
- **真实环境模拟**：在尽可能接近生产环境的硬件、操作系统、网络和数据库中执行。
- **全面性**：不仅测试功能，还包括性能、安全、稳定性、兼容性等非功能性指标。

#### 系统测试的主要类型

- 
  **功能测试** 软件的每个按钮、每个流程是否按预期工作。

- **性能测试** 在高并发或大数据量下，系统的响应速度和资源消耗（JMeter, LoadRunner）。
- **安全测试** 验证系统是否有漏洞（SQL注入、XSS等），保护数据安全。
- **兼容性测试** 在不同浏览器（Chrome/Safari）、不同手机系统（iOS/Android）下的表现。
- **健壮性/可靠性** 系统在断网、断电或输入错误数据时的自我恢复能力。
- **回归测试** 修复 Bug 后，重新测试相关功能确保没有引入新错误。

#### 常用工具

- **功能自动化**：Selenium (Web端)、Appium (移动端)。
- **性能压力**：Apache JMeter、Locust。
- **安全扫描**：Burp Suite、OWASP ZAP。
- **缺陷管理**：Jira、禅道 (Zentao)、PingCode。
- **兼容性云平台**：BrowserStack、LambdaTest。

### 验收测试

软件开发生命周期的**最后一道关口**。

在经历单元、集成、系统测试后，验收测试的唯一目的就是：**确认软件是否达到了交付标准，以及用户是否愿意为此“买单”**。它不再是为了找 Bug，而是为了“建立信心”。

#### 核心特征

- **业务导向**：侧重于真实业务场景，而非技术细节（比如不关心数据库字段，只关心报表数据是否准确）。
- **最后通牒**：测试结果直接决定项目是 **GO（上线）** 还是 **NO GO（返工）**。
- **法律/合同效力**：验收通过通常意味着项目尾款的支付或责任的移交。

#### 验收测试的评价标准

验收通过不仅看功能，还要检查以下**交付物**：

1. **功能完整性**：核心业务流 100% 跑通。
2. **遗留缺陷**：所有致命（Critical）和严重（Major）级别的 Bug 已关闭。
3. **文档交付**：用户手册、安装指南、维护说明等是否齐全。
4. **非功能指标**：在真实带宽和硬件环境下，性能是否达标。

#### 常用辅助工具

由于验收测试强调业务流程，使用的工具多偏向于**协作与记录**：

- **业务流程录制**：Loom 或各种屏幕录制软件，用于留存证据。
- **协作管理**：**Jira** (用于最终确认列表)、**Confluence** (存储验收标准文档)。
- **自动化辅助**：Robot Framework (由于其基于关键字驱动，业务人员也易于看懂测试逻辑)。

## 程序可见度

 是否查看代码内部逻辑（黑盒关注功能，白盒关注逻辑）。

### 黑盒测试

测试者将受测软件视为一个不透明的“黑盒子”，在完全不考虑程序内部结构和逻辑代码的情况下，仅依据需求规格说明书，通过在软件界面进行输入和输出来检查功能是否符合预期。

### 白盒测试

一种基于程序**内部结构和源代码**的软件测试方法。测试者需了解程序的架构、设计及具体的代码实现，通过检查内部逻辑路径、条件分支、循环结构等来验证程序是否按设计要求正确运行

### 灰盒测试

介于黑盒测试与白盒测试之间的一种软件测试方法。测试者对被测软件的内部结构、数据结构或算法有**部分了解**，但不会像白盒测试那样深入到每一行代码的逻辑，而是利用这些内部知识来辅助设计更精准的外部功能测试。

它通常被视为一种“半透明”的测试，既关注用户界面的输入输出（黑盒特性），又关注模块间的接口、数据库交互和内部架构（白盒特性）。

## 标准化测试流程

一个完整的测试周期通常包括以下关键步骤：

1. **需求分析**：参与评审，确保对功能点的理解与开发/产品一致。
2. **测试计划**：确定测试范围、资源、时间表及风险评估。
3. **测试用例设计**：编写详细的输入、执行步骤和预期结果。
4. **环境搭建**：配置测试所需的硬件、网络及数据库。
5. **执行测试**：运行用例，通过 Bug管理工具（如 Bugzilla） 记录缺陷。
6. **缺陷回归**：修复后重新测试，确保 Bug 已解决且未引入新问题。
7. **测试评估/总结**：分析测试结果，出具测试报告以决定是否发布。
